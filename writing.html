The Cost of Abstraction

March 2026

Abstractions are the fundamental tool of software engineering. They allow you to reason about complex systems by hiding irrelevant detail. Without them, nothing of any scale would be possible. This is not in dispute.

What is in dispute is whether the abstractions you are using are hiding the right things, from the right people, at the right time.

Consider a container orchestration platform. It abstracts away the question of which physical machine runs your workload. This is useful: you can scale, reschedule, and replace nodes without rewriting your application. The abstraction earns its cost. But the same platform also abstracts away networking, storage, scheduling, and security policy. Each layer hides complexity. Each layer also hides failure modes.

When something goes wrong — and it will go wrong — the abstraction that protected you from complexity now stands between you and the diagnosis. The question is not whether to use abstractions. The question is whether you understand what is underneath them well enough to reason about failures.

An abstraction you cannot see through is a liability, not an asset, when the system misbehaves.

The homelab is a partial answer to this problem. Running a firewall on bare metal, configuring VRRP by hand, managing connection state synchronisation explicitly — these are not efficient uses of time compared to clicking through a vendor GUI. But they produce engineers who understand what the GUI is doing. That understanding is not academic. It is the difference between a 30-minute outage and a 3-hour one.

The practical question to ask of any abstraction is: when this breaks, will I be able to reason about why? If the answer is no, you have accepted a risk. You may have accepted it knowingly and correctly. But you should have accepted it knowingly.

High Availability is a Trade-off, Not a Feature

February 2026

High availability is sold as a feature. It is not. It is a trade-off. The trade is: you accept more complexity, more maintenance burden, and new categories of failure in exchange for reduced exposure to a specific class of failure — single-node downtime.

This trade is worth making in many production contexts. It is not automatically worth making everywhere, and it is almost never worth making without understanding what you are trading.

Consider a two-node VRRP setup with keepalived and conntrackd, as documented in the oak03ufw01 device page. The configuration provides failover in under 3 seconds when the primary node fails. Established connections survive. This is genuinely useful.

But the same configuration introduces split-brain scenarios. If the synchronisation network between the two nodes fails while both nodes remain operational, both claim the virtual IP. Traffic becomes unpredictable. Resolving split-brain requires manual intervention. The monitoring must detect it. An operator must investigate. The STONITH mechanisms that prevent split-brain in enterprise HA clusters require out-of-band management hardware that the Sophos appliances do not have.

The question is not whether HA is good or bad. The question is: what failure modes does this configuration introduce, and are they less severe than the failure modes it eliminates?

Failure mode eliminated Failure mode introduced
Single-node hardware failure Split-brain (both nodes claim VIP)
Single-node OS crash Configuration drift between nodes
Single-node maintenance window Increased maintenance surface (two nodes)
Single-node network failure Sync network failure (new dependency)

In a home lab, the educational value of understanding HA justifies the complexity. In a production environment, the calculus depends on your specific availability requirements, your operational maturity, and your ability to monitor and respond to the new failure modes. HA that is not regularly tested is not HA. It is the illusion of HA.

Before implementing HA, ask: when was the last time we tested failover? If the answer is never, or more than six months ago, you do not have HA. You have a second node that has never been proven to work.

The Case for Running End-of-Life Hardware

January 2026

End-of-life hardware receives no security patches. This is a real risk and should not be dismissed. If you are running EOL hardware in a context where it is exposed to untrusted networks or handles sensitive data without compensating controls, you are making a poor decision.

That said, the reflexive dismissal of EOL hardware as universally unsuitable is also wrong. It conflates risk with context. The relevant questions are: what is the threat model, what compensating controls exist, and what is the value of running the hardware?

The homelab runs a Power Mac G4 (PowerPC, 2001), an SGI Indy (MIPS R4600, 1993), and a Sun Ultra 10 (SPARC IIi, 1999). These machines receive no vendor support. They run software that is decades old. They are also isolated on a dedicated VLAN with no inbound access from untrusted networks. The risk is managed, not eliminated.

The value is concrete. Writing software that runs on PowerPC, MIPS, and SPARC forces you to confront assumptions that x86 development conceals:

  • Endianness is not a given. Big-endian architectures expose bugs in code that assumes little-endian byte order.
  • Memory alignment matters. RISC architectures fault on unaligned memory access. x86 handles it in hardware, masking the bug.
  • Instruction set constraints are real. Code that runs efficiently on a modern out-of-order x86 core may perform poorly on an in-order MIPS pipeline.
  • Compiler assumptions are visible. Software that assumes 64-bit pointers, or assumes a particular calling convention, fails loudly on these platforms.

Running diverse hardware is a form of adversarial testing. It surfaces assumptions. Surfaced assumptions can be corrected. Hidden assumptions become production incidents.

This is also why this website works on browsers from the year 2000. Testing on Netscape Navigator 4 and Internet Explorer 5 is not nostalgia. It is a discipline that forces you to separate content from presentation, and to build systems that degrade gracefully rather than failing completely.

Monitoring Is Not Debuggability

December 2025

Monitoring tells you that something is wrong. Debuggability tells you why. These are different problems, and conflating them produces systems that are well-monitored but poorly understood.

A Prometheus alert fires. CPU usage on a node is at 95%. The alert is correct. The monitoring is working. Now what? You SSH into the node. You run top. You see a process consuming CPU. You kill it. The alert resolves. Two hours later, the alert fires again.

This is not a monitoring problem. This is a debuggability problem. The monitoring told you the symptom. It did not tell you the cause. Without understanding the cause, the intervention was palliative, not corrective.

Debuggability requires different investments than monitoring. Monitoring is about aggregation and alerting. Debuggability is about context, correlation, and causation. Specifically:

  • Structured logging with correlation IDs allows you to trace a request across services. Without correlation IDs, logs from different services are unconnected data points.
  • Meaningful error messages include context: what was being attempted, what state the system was in, what the expected and actual values were. "Connection refused" is a symptom. "Connection to database replica at 10.0.10.5:5432 refused after 3 retries; primary at 10.0.10.4:5432 is reachable" is debuggable.
  • Documented failure modes allow operators to match observed symptoms to known causes. This is not glamorous work. It is the difference between a 20-minute diagnosis and a 4-hour one.
  • Runbooks encode institutional knowledge. They are not a substitute for understanding, but they prevent the same diagnosis from being performed from scratch every time.

The question to ask of any system you operate is: if this system fails at 3am, can the on-call engineer diagnose the cause without domain expertise? If the answer is no, the system is not sufficiently debuggable, regardless of how many dashboards you have.

Progressive Enhancement Is an Argument About Values

November 2025

Progressive enhancement is a web development technique. It is also an argument about values. The technique is: build a baseline that works everywhere, then add layers of enhancement for browsers that support them. The argument is: the web should be accessible to everyone, not just those with modern hardware and fast connections.

The technique is not difficult. HTML that is semantically correct and does not depend on JavaScript to render content works on any browser that can parse HTML. This includes browsers from 1995. It includes text-mode browsers like Lynx. It includes screen readers. It includes search engine crawlers. The baseline is free, or close to it, if you build it first.

The reason most websites do not do this is not technical. It is cultural. The default in modern web development is to start with a JavaScript framework and add content. The baseline is JavaScript-rendered HTML. Remove JavaScript, and you get a blank page. This is not a technical constraint. It is a choice, made implicitly, to exclude users who cannot or do not run JavaScript.

A website that requires JavaScript to display text has decided that some users do not deserve to read it.

This is not an argument against JavaScript. JavaScript enables genuinely useful things: interactive interfaces, real-time updates, offline capability. The argument is about sequencing. Build the content layer first. Build it so it works without anything else. Then add JavaScript as an enhancement, not a prerequisite.

The same principle applies beyond the web. Systems that degrade gracefully under partial failure are more reliable than systems that fail completely. APIs that return useful partial responses when some dependencies are unavailable are more useful than APIs that return errors. The pattern is general: design for the degraded case first, then enhance.
