Overview

oak02ufw01 is an Ubuntu-based firewall running on Sophos SG 210 rev.3 hardware. It replaced the EOL Sophos UTM firmware. The device provides routing, firewall, and VPN services for the lab network.

Motivation

The Sophos SG 210 reached EOL in 2021. No security updates. The hardware remained functional. Replacing the proprietary firmware with Ubuntu provided current security patches and full control over configuration.

This deployment served as a learning exercise. Commercial firewall appliances abstract configuration behind web interfaces. Building a firewall from Ubuntu exposed the underlying mechanisms: iptables/nftables, routing tables, network namespaces, and kernel parameters.

The goal was understanding how firewalls work, not just how to configure them.

Installation

Ubuntu Server 24.04 LTS installed via USB. The Sophos hardware uses standard BIOS. Boot from USB, select the mSATA SSD as the installation target. The installer detected all 8 network interfaces.

Network interface naming used predictable names (enp2s0, enp3s0, etc.). Netplan configuration mapped interfaces to VLANs. The WAN interface connected to the ISP router. LAN interfaces connected to the managed switch.

Network Configuration

The firewall bridges multiple VLANs. Netplan defines bridge interfaces. Each VLAN gets a bridge. The firewall acts as the default gateway for each VLAN.

Example Netplan configuration for VLAN 20:

network:
  version: 2
  ethernets:
    enp2s0:
      dhcp4: false
  bridges:
    br20:
      interfaces: [enp2s0]
      addresses: [10.0.20.1/24]
      parameters:
        stp: false

IP forwarding enabled via sysctl. The kernel routes packets between VLANs based on firewall rules.

Firewall Rules

nftables replaced iptables. The ruleset uses tables, chains, and rules. The configuration file lives at /etc/nftables.conf.

Default policy: drop all traffic. Explicit allow rules for permitted traffic. Stateful connection tracking reduces rule complexity. Established connections bypass most rules.

Example rule allowing VLAN 20 to access the internet:

table inet filter {
  chain forward {
    type filter hook forward priority 0; policy drop;
    
    iifname "br20" oifname "wan0" ct state new,established,related accept
    iifname "wan0" oifname "br20" ct state established,related accept
  }
}

Inter-VLAN rules restrict traffic. Management VLAN cannot be reached from other VLANs. IoT VLAN has no inbound access.

NAT Configuration

Source NAT (SNAT) translates internal IPs to the WAN IP. nftables handles this in the nat table:

table inet nat {
  chain postrouting {
    type nat hook postrouting priority 100;
    oifname "wan0" masquerade
  }
}

Port forwarding uses destination NAT (DNAT). External ports map to internal services. Example: external port 2222 forwards to internal SSH on port 22.

VPN

WireGuard provides VPN access. Configuration at /etc/wireguard/wg0.conf. The VPN subnet is 10.0.100.0/24. Clients receive IPs via static assignment.

VPN traffic routes through the firewall. Clients access internal VLANs based on firewall rules. Split tunneling sends only internal traffic through the VPN. Internet traffic uses the client's local connection.

DNS

Unbound runs on the firewall. It provides DNS for internal VLANs. Local zone maps internal hostnames. Upstream queries forward to Cloudflare DNS over TLS.

Ad blocking uses a blocklist. Unbound returns NXDOMAIN for blocked domains. The list updates weekly via cron.

Monitoring

Prometheus node_exporter exposes system metrics. Grafana visualizes CPU, memory, network throughput, and connection counts. Alerts trigger on high CPU or connection table exhaustion.

Logs go to journald. Critical events forward to a central syslog server. Log rotation prevents disk exhaustion.

Challenges

Network interface naming was inconsistent across reboots initially. Netplan configuration used MAC addresses to ensure stable naming. This fixed the issue.

nftables syntax differs from iptables. The transition required reading the nftables wiki and testing rules in a VM before deploying to production.

Connection tracking table size needed tuning. Default limits were too low for the number of concurrent connections. Increased net.netfilter.nf_conntrack_max via sysctl.

Lessons Learned

Building a firewall from scratch clarifies what commercial appliances abstract. NAT, routing, and firewall rules are kernel features. The appliance provides a configuration interface and management tools.

Ubuntu provides the same capabilities. The tradeoff is manual configuration versus vendor support. For a lab environment, the learning value outweighs the convenience of a turnkey solution.

Documentation matters. Runbooks for common tasks reduce troubleshooting time. Configuration files need comments explaining non-obvious choices.
