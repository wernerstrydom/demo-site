Overview

oak03ufw01 is a secondary Ubuntu-based firewall running on Sophos SG 210 rev.3 hardware. It serves as a testbed for high availability configurations and failover scenarios.

Motivation

Single firewall creates a single point of failure. Testing high availability requires two devices. oak03ufw01 provides the second node for HA experiments.

The goal is understanding how commercial HA solutions work. VRRP, keepalived, and connection state synchronization are concepts. This device makes them concrete.

Configuration

The base configuration mirrors oak02ufw01. Same Ubuntu version, same network layout, same firewall rules. The differences are hostname, IP addresses, and HA-specific configuration.

This symmetry is intentional. HA requires identical configurations except for node-specific values. Testing configuration management tools (Ansible) benefits from this setup.

High Availability Setup

keepalived provides VRRP-based failover. Both firewalls run keepalived. One is MASTER, the other is BACKUP. They share a virtual IP (VIP) that floats between them.

Example keepalived configuration:

vrrp_instance VI_1 {
    state BACKUP
    interface br20
    virtual_router_id 51
    priority 90
    advert_int 1
    
    virtual_ipaddress {
        10.0.20.1/24
    }
}

oak02ufw01 has priority 100 (MASTER). oak03ufw01 has priority 90 (BACKUP). When oak02ufw01 fails, oak03ufw01 assumes the VIP and becomes MASTER.

Connection State Synchronization

conntrackd synchronizes connection tracking state between firewalls. When failover occurs, established connections continue without interruption.

Configuration at /etc/conntrackd/conntrackd.conf. The firewalls exchange state over a dedicated sync interface. UDP multicast carries state updates.

This works for stateful connections. Stateless protocols (UDP without connection tracking) don't benefit from synchronization.

Testing Failover

Testing involves stopping keepalived on the MASTER. The BACKUP detects the failure within 3 seconds (3x advert_int). It assumes the VIP and starts forwarding traffic.

Established SSH connections survive failover if conntrackd is working. New connections work immediately. The failover is transparent to clients.

Testing also includes network cable disconnection, power failure simulation, and kernel panics. Each scenario validates different failure modes.

Challenges

Split-brain scenarios occur when both nodes think they are MASTER. This happens if the sync network fails but both nodes remain operational. Both nodes claim the VIP. Routing becomes unpredictable.

Fencing mechanisms prevent split-brain. STONITH (Shoot The Other Node In The Head) forcibly powers off the failed node. This requires IPMI or similar out-of-band management. The Sophos hardware lacks IPMI.

Without fencing, split-brain requires manual intervention. Monitoring detects the condition. An operator investigates and resolves it.

conntrackd state synchronization has overhead. High connection rates increase CPU usage. The Atom CPU handles moderate loads but struggles under stress testing.

Lessons Learned

HA adds complexity. Two devices require twice the maintenance. Configuration drift between nodes breaks failover. Automation (Ansible) reduces drift but doesn't eliminate it.

Testing failover regularly matters. Untested HA is unreliable HA. Scheduled failover tests validate the configuration and build confidence.

HA doesn't eliminate all failure modes. It handles node failures. It doesn't handle configuration errors, software bugs, or upstream network failures. HA is one tool, not a complete solution.

For a home lab, HA is educational but not essential. The learning value justifies the complexity. For production, HA is a requirement, not an option.

Future Work

Testing active-active configurations where both firewalls handle traffic simultaneously. This requires more complex routing and load balancing.

Integrating with monitoring to automatically trigger failover based on health checks, not just keepalived advertisements.

Exploring alternatives to keepalived: pacemaker, corosync, or custom scripts.
