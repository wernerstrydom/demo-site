Role and rationale

oak03ufw01 is the secondary firewall. In normal operation, it holds no virtual IPs and forwards no traffic. Its purpose is to assume the gateway role if oak02ufw01 fails.

A single firewall is a single point of failure. For a homelab, a single point of failure is acceptable — downtime has no business consequence. The reason to run a second firewall is educational: understanding how HA failover works requires a working HA setup. oak03ufw01 makes the concepts of VRRP, connection state synchronisation, and split-brain concrete rather than theoretical.

This is the correct reason to run HA in a homelab. The incorrect reason is to claim production-grade availability without understanding the failure modes HA introduces. See High Availability is a Trade-off, Not a Feature.

VRRP configuration

vrrp_instance VI_MGMT {
    state BACKUP
    interface enp1s0
    virtual_router_id 10
    priority 90
    advert_int 1
    virtual_ipaddress {
        10.0.10.1/24
    }
}

vrrp_instance VI_PROD {
    state BACKUP
    interface enp1s0.20
    virtual_router_id 20
    priority 90
    advert_int 1
    virtual_ipaddress {
        10.0.20.1/24
    }
}

When oak02ufw01 stops sending VRRP advertisements, oak03ufw01 detects the absence within 3 seconds and assumes the virtual IPs. It begins forwarding traffic immediately. Established connections are preserved if conntrackd state synchronisation is current.

Split-brain risk

Split-brain occurs when both nodes believe they are the VRRP master. This happens if the sync link (enp1s1) fails while both nodes remain operational. Both nodes claim the virtual IPs. ARP responses from two different MAC addresses for the same IP cause unpredictable routing on the network.

Detection: monitor for duplicate ARP responses on the network. Prometheus with an ARP monitoring exporter can detect this condition. Resolution requires manual intervention: determine which node is the intended master, stop keepalived on the other, and verify the sync link.

Prevention via STONITH (fencing) is not available on this hardware. The Sophos SG 210v3 has no IPMI or out-of-band management interface. This is a known limitation. The risk is accepted for a lab environment. For production, hardware with IPMI is required for reliable HA.

Failover testing

Failover is tested monthly. Procedure:

  1. Verify oak03ufw01 is operational: systemctl status keepalived conntrackd.
  2. Stop keepalived on oak02ufw01: systemctl stop keepalived.
  3. Verify oak03ufw01 assumes VIPs: ip addr show should show the virtual IPs on oak03ufw01 within 3 seconds.
  4. Verify connectivity: ping 10.0.20.1 from a Production VLAN device. Establish an SSH session before failover and verify it survives.
  5. Restart keepalived on oak02ufw01. Verify it preempts oak03ufw01 and resumes as master.
  6. Document the test date and result.

Untested HA is not HA. It is a second node that has never been proven to work. Monthly testing is the minimum acceptable interval.

Lessons from operating this setup

Configuration drift between the two nodes is the most common source of problems. When a firewall rule is added to oak02ufw01, it must be added to oak03ufw01. Automation (Ansible) reduces drift but does not eliminate it. The Ansible playbook must be applied to both nodes, and the application must be verified.

conntrackd state synchronisation adds CPU overhead. Under normal load, the Atom C2358 handles it without issue. Under stress testing (high connection rates), CPU usage increases noticeably. This is not a problem in practice but is worth monitoring.

The 3-second failover time is acceptable for most use cases. SSH sessions survive. HTTP connections that use TCP keepalives survive. Short-lived connections (DNS, UDP) may fail and retry. Applications that do not handle transient failures gracefully will exhibit errors during failover. This is a property of the application, not the HA setup.
