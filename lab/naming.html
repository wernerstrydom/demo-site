Format

Hostnames follow the pattern: location-number-function-sequence

Example: oak02ufw01

  • oak - Location code (Oakland)
  • 02 - Rack or zone number
  • ufw - Function code (Ubuntu firewall)
  • 01 - Sequence number for this function

Location Codes

Code Location Description
oak Oakland Primary lab location

Function Codes

Code Function Description
ufw Ubuntu Firewall Firewall/router running Ubuntu
esxi ESXi Host VMware ESXi hypervisor
pve Proxmox VE Proxmox virtualization host
nas NAS Network attached storage
sw Switch Network switch
ap Access Point Wireless access point

Rationale

This naming scheme provides immediate context about a device without requiring documentation lookup. The location code matters when you have multiple sites or plan to expand. The rack number helps with physical troubleshooting. The function code indicates the device's role. The sequence number distinguishes multiple devices with the same function.

The scheme is deterministic. Given a device's purpose and location, the hostname is predictable. This reduces cognitive load when managing infrastructure.

Examples

Hostname Meaning
oak02ufw01 Oakland, rack 2, Ubuntu firewall, first instance
oak02ufw02 Oakland, rack 2, Ubuntu firewall, second instance
oak01esxi01 Oakland, rack 1, ESXi host, first instance
oak03nas01 Oakland, rack 3, NAS, first instance

DNS Configuration

Internal DNS uses the .lab TLD. Full hostnames: oak02ufw01.lab

DNS A records point to management interfaces. CNAME records provide service-based aliases when needed.

Alternatives Considered

Random names (mythology, planets, etc.) are memorable but don't scale. They require documentation to understand a device's purpose. They work for small deployments but break down with growth.

UUIDs or random strings are unique but not human-readable. They require constant documentation lookup.

Descriptive names like "firewall-primary" don't handle multiples well and lack location context.

The structured approach trades memorability for predictability and scalability.
