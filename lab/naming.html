Device naming is a recurring topic in homelab and professional engineering communities. The choice of naming scheme is not merely aesthetic. It affects how quickly you can identify a device, understand its role, and locate it physically. A good naming scheme encodes useful information without requiring a lookup table.

Scheme structure

Hostnames follow this pattern:

{site}{rack}{type}{seq}
Component Length Description Example
site 3 chars Physical location identifier. Lowercase alpha. Derived from a location name. oak (Oakland)
rack 2 digits Rack or physical grouping number. Zero-padded. 01–99. 02
type 3 chars Device type abbreviation. Lowercase alpha. Encodes the primary role. ufw (firewall)
seq 2 digits Sequence number within the type at that site/rack. Zero-padded. 01–99. 01

Full example: oak02ufw01 = Oakland, rack 02, firewall, unit 01.

Type abbreviations

Abbreviation Role Notes
ufw Firewall From "uncomplicated firewall" (ufw), the Ubuntu firewall tool. Also a common abbreviation for network firewall.
srv General server Hypervisors, storage servers, and general-purpose compute.
dns DNS resolver Internal DNS and filtering.
mon Monitoring Prometheus, Grafana, alerting.
git Git server Internal source control.
nas Network-attached storage ZFS-backed storage nodes.

Rationale

The naming scheme is designed to answer three questions at a glance:

  1. Where is it? The site prefix identifies the physical location. In a multi-site lab, this prevents confusion between identically-named devices at different locations.
  2. What is it? The type abbreviation identifies the primary role. You do not need to look up oak02ufw01 to know it is a firewall.
  3. Which one? The rack and sequence numbers distinguish between multiple devices of the same type. oak02ufw01 and oak03ufw01 are both firewalls, in different racks, both the first firewall in their respective rack.

What the scheme does not encode

The scheme deliberately does not encode:

  • IP addresses. IP addresses change. Encoding them in hostnames creates maintenance burden and misleads when addresses are reassigned. Use DNS.
  • Hardware platform. Hardware is replaced. A hostname that encodes the hardware type becomes incorrect when the hardware changes. The device type (role) is stable; the hardware is not.
  • Operating system. Same reasoning as hardware. The role is stable; the OS may change.

Common alternatives and their trade-offs

Scheme Example Advantage Disadvantage
Sequential numbers server01 Simple. No encoding to remember. Tells you nothing. Requires a lookup for every device.
Mythological names zeus, athena Memorable. No confusion with functional names. Tells you nothing. Does not scale. Runs out of memorable names.
Role + number firewall01 Role is clear. Simple. No location encoding. Ambiguous in multi-site environments.
This scheme oak02ufw01 Encodes site, rack, role, and sequence. Scales. Less memorable than names. Requires learning the abbreviations.

No naming scheme is universally correct. The right scheme depends on the scale of your environment, the number of people who need to use it, and how often you need to identify devices without access to documentation. The scheme documented here is appropriate for a single-site or small multi-site homelab with one operator. A larger environment would benefit from a more formal CMDB and a scheme that integrates with it.

Opinion: Mythological or random-word naming schemes are fine for small personal environments where you know every machine by name. They fail at scale and when onboarding new operators. If you cannot tell what a machine does from its name, you have added a lookup step to every interaction with it. That cost is small per interaction and large in aggregate.
